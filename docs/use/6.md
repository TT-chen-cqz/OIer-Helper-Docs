# 使用指南

本章节详细说明如何高效使用 OIer-Helper，包括任务识别逻辑、提问技巧、输出格式说明和注意事项。

## 任务识别逻辑

助手接收到消息后，会按以下优先级判断任务类型：

1. **代码调试（任务 C）**：同时出现题目和代码 → 优先调试。
2. **题目分析（任务 A）**：出现题目链接、题号或明确要求分析题面。
3. **算法讲解（任务 B）**：出现算法名称，且无题目或代码。
4. **题目推荐（任务 D）**：出现“推荐”“找题”等搜索意图。

> 若您的请求模糊，助手可能询问补充信息。请尽量明确表达需求。

## 提问技巧

| 目标 | 推荐提问方式 |
|------|-------------|
| 分析某道题 | `https://...` 或 `洛谷 P3371 这题怎么做？` |
| 讲解算法 | `讲一下 KMP 算法` 或 `线段树模板` |
| 调试代码 | `我的代码（粘贴）哪里错了？题目是 https://...` |
| 推荐题目 | `推荐几道树形 DP 的题` 或 `有没有 Codeforces 上的贪心练习题？` |
| 获取模板 | `给一个堆优化 Dijkstra 的模板`（此情况通常归入算法讲解，但会侧重代码） |

## 回答格式说明

### 题目分析输出结构
- 标题分级清晰，包含“题目判断”“解法思路”“关键观察”“算法步骤”“正确性说明”“复杂度分析”“C++17 参考代码”“易错点”“参考来源”。
- 代码块使用 ```cpp ... ```，并遵循项目代码规范。
- 若未找到题解，会在开头声明，并附带“测试提醒”段落。

### 算法讲解输出结构
- 包含“算法名称”“适用场景”“核心思想”“算法流程”“正确性证明”“复杂度分析”“C++17 模板代码”“模板使用说明”“常见变形”“易错点”“练习题推荐”“参考资料”。

### 调试输出结构
- 先简要说明题目核心解法，然后列出“标准思路关键点”，接着“你的代码整体判断”，再逐条列出“主要错误”（位置、原因、后果、修改方式），最后给出“修改建议”“修正版代码”“建议测试数据”。

### 题目推荐输出结构
- 以表格列出推荐题目（#、平台、题号、标题、难度、链接），附带简要说明和练习建议。
- 默认不附题解，若用户要求可增加简短思路提示。

## 代码规范

所有生成的 C++ 代码遵循以下规范：

- 使用 `#include <bits/stdc++.h>`。
- 主函数 `int main()`，并开启 `ios::sync_with_stdio(false); cin.tie(nullptr);`。
- 不使用 `#define int long long`（除非特别说明）。
- 需要 64 位整数时使用 `long long` 或 `using ll = long long;`。
- 数组大小根据题目数据范围动态或静态设置。
- 变量命名清晰，关键部分添加注释。
- 正确处理多组数据（多测清空）。
- 注意 1-indexed 与 0-indexed 选择。
- INF 使用足够大值（如 `1e18`）。

## 注意事项

- **诚实原则**：助手不会编造题解来源，也不会声称代码已通过评测。未找到可靠题解时会明确说明。
- **版权与学术诚信**：助手总结公开思路，不复制他人完整代码。生成代码时会提醒用户遵守社区规则，不要直接提交作弊。
- **不编译不运行**：助手不会尝试编译或运行任何代码，所有代码仅供参考，用户需自行测试。
- **联网依赖**：题目分析、算法讲解和题目推荐需要联网搜索，请确保网络畅通。
- **若响应不理想**：可尝试重新措辞，或提供更多上下文（如题目数据范围、时间限制、错误类型等）。
